Análisis Estático con SonarQube en Jenkins

11 Mar 2017 . categoría: integration . Comentarios
#ci #cqm #ccq #jenkins #sonarqube

Una de las actividades que se consideran relevantes durante el ciclo de desarrollo de Software es la revisión de código, que puedes hacer en modo estático (sin ejecutar el código fuente) y en modo dinámico (durante la ejecución del código fuente). El Análisis Estático lo puedes relacionar con el visto bueno que el desarrollador le dá a su trabajo antes de promoverlo a su utilización, es decir revisión de errores potenciales, vulnerabilidades, impacto en calidad del diseño (Code Smells), estándares de codificación, por mencionar algunos. Si esta revisión la delegas a un herramienta a la cual le especificas reglas de revisión, puedes lograr un concepto de Calidad de Codigo Continua.

 

  • Intención
  • Motivación
  • Aplicabilidad

Realizar la configuración de una tarea de análisis de código en Jenkins, utilizando la herramienta SonarQube.

Imagina un proceso de desarrollo o de pruebas en donde una de las precondiciones para continuar con el flujo es la revisión y aceptación del código fuente. La revisión del código podría ser una tarea laboriosa y dependiente de la pericia, disposición, observación y exactitud del o los desarolladores.

Sin embargo la madurez, estabilización y aumento de funcionalidad implica un crecimiento considerable del código fuente, lo cual imposibilita la tarea de revisión, se convierte en desgastante y poco productiva. Una herramienta que ayude a la automatización de revisión de código, permitiría ahorrar el dedicar tiempo de los desarrolladores en análisis.

  • Cuando un desarrollador requiera evaluar la calidad de su código posterior a la terminación de un cambio.
  • Cuando se requiera realizar una actividad de re-estructuración de código fuente.
  • Cuando se tenga un proceso automático de integración o de pruebas que requiera como insumo un estatus aceptable de la calidad del código.

 

Estructura

  1. El equipo de desarrollo realiza cambios en el código fuente.
  2. Se ejecuta el proceso de Sonar-Scanner para evaluar la calidad del código a partir de un Análisis Estático; Jenkins es un buen candidato a ejecutar dicho proceso.
  3. El proceso Sonar-Scanner trabaja en conjunto con el servidor SonarQube y se envían los resultados del análisis.
  4. Las incidencias detectadas durante el análisis, visualizadas en SonarQube, son evaluadas por el equipo de desarrollo.
  5. Según el grado de importancia de las incidencias y el nivel de calidad alcanzado respecto al definido, el equipo se encarga de dar el mantenimiento requerido (volver a paso inicial).

 

Implementación

Prerrequisitos

  1. (Opcional) Tener configurado un servidor de base de datos (Microsoft SQL Server, MySQL, Oracle, PostgreSQL); por default se puede utilizar la base de datos H2 que trae embebida SonarQube.
  2. Tener instalado el JDK (compatible con la versión de SonarQube server)
  3. Tener configurada un servidor de SonarQube.
  4. Tener configurado el ambiente de ejecución de Sonar-Scanner.
  5. Tener instalado el plugin para Jenkins SonarQube Scanner.
  6. (Opcional) Tener cuenta en GitHub o en SonarQube.
Tip:
Puedes omitir la configuración de base de datos y utilizar la embebida H2 que tiene por default SonarQube, solo ten cuenta que esa base de datos se ocupa para fines de pruebas.
Importante: Ambiente
Ubuntu 16.04.1 LTS alojado en DigitalOcean.
JDK 1.8
SonarQube 6.2
SonarQube Scanner 2.8

 

Configuración de servidor SonarQube

El servidor de SonarQube es un administrador que te muestra el estado de los proyectos respecto a: errores, vulnerabilidades, Deuda Técnica, código malhecho, código duplicado, por mencionar las principales características. También puedes ver gráficos con estadísticos de los proyecto, control de usuarios (seguridad), configuración de plugins (necesarios si se desea extender el catálogo de analizadores por lenguage de programación utilizado), administración de reglas de codificación, perfiles de calidad, entre otras cosas.

  • La instalación consiste en descargar, descomprimir y colocar distribución de SonarQube en el directorio para paquetes independientes del sistema.

wget https://sonarsource.bintray.com/Distribution/sonarqube/sonarqube-6.2.zip

unzip sonarqube-6.2.zip
sudo mv sonarqube-6.2 /opt/sonar/sonarqube/6.2


  • Editar archivo de propiedades para configuración de base de datos, servidor Web, motor de procesamiento, configuración Elasticsearch, centro de actualización, Logging, por mencionar las más importantes. En general se suele configurar el apartado para conexión a base de datos (sino se quiere utilizar la H2 embebida), así como también se edita el puerto y contexto por default para exponer el servidor. Para el ejemplo utilizaremos la base de datos embebida y todos los valores por default.

sudo vim /opt/sonar/sonarqube/6.2/conf/sonar.properties 


#sonar.jdbc.username=
#sonar.jdbc.password=

# El valor vacío indica que utilizarás la base de datos embebida H2
#sonar.jdbc.url=


# El valor por default es vacío (raíz del contexto)
#sonar.web.context=/sonar

# El valor por default es 9000
#sonar.web.port=


  • Hablando del control del servidor SonarQube se tienen ejecutables para diferentes plataformas (Windows, Linux, MacOS) con arquitecturas x86 de 32 y 64 bits. Las funciones disponibles son:
    • console para inicializar y visualizar el log de ejecución del servidor
    • start para inicializar el servidor
    • stop para detener la ejecución del servidor
    • restart para reiniciar el servidor
    • status para conocer el estatus de ejecución del servidor
    • dump para hacer una operación de mantenimiento sobre el servidor

sudo /opt/sonar/sonarqube/6.2/bin/linux-x86-64/sonar.sh start


  • Para comprobar que tu servidor SonarQube esté funcionando correctamente, accede a la dirección http://localhost:9000/ (si no cambiaste los valores default para el servidor). Se debe de mostrar una vista parecida a la siguiente:

 

Configuración del escáner de SonarQube

El SonarQube Scanner es el inicializador predeterminado para el análisis de los projectos con SonarQube.

Importante:
Considera que según la versión de SonarQube es la versión mínima que necesitas para tu JDK. El mensaje de error al respecto, que aparece durante la ejecución es parecido a: Exception in thread “main” java.lang.UnsupportedClassVersionError: org/sonarsource/scanner/cli/Main : Unsupported major.minor version 5X.0
  • Descargar, descomprimir y colocar distribución de Sonar-Scanner.

wget https://sonarsource.bintray.com/Distribution/sonar-scanner-cli/sonar-scanner-2.8.zip

unzip sonar-scanner-2.8.zip
sudo mv sonar-scanner-2.8 /opt/sonar/sonar-scanner/2.8


  • Editar archivo de propiedades para asignar URL de servidor SonarQube, codificación de caracteres y conexión a base de datos. Al tener configurado un servidor SonarQube, solo es necesario agregar la URL (en el caso de no alcanzar al servidor SonarQube por localhost).

sudo vim /opt/sonar/sonar-scanner/2.8/conf/sonar.properties 


# Servidor de SonarQube, el default es localhost
#sonar.host.url=http://localhost:9000

 

SonarQube y Jenkins en acción

Para poder ejecutar SonarQube desde Jenkins en necesario asociar tanto el servidor como el ambiente de ejecución.

  • El acceso al servidor requiere de un token de autenticación o que el acceso anónimo este desactivado. Dado que no existe certeza de la segunda premisa, se tiene que configurar el token en Usuario > My Account > Security, donde se establece un nombre del token y se copia token generado (el cual solo aparecerá una vez).

  • Tener configurado el servidor te permite inyectar valores a las propiedades de ejecución del análisis, por ejemplo: URL, token de autenticación, usuario, contraseña, datos de conexión a base de datos. Dicha configuración se realiza en Jenkins > Home > Manage Jenkins > Configure System en el apartado SonarQube servers. Los dos valores más importantes son: la Server URL y el Server Token Authentication.

  • Posterior a la configuración del servidor SonarQube, se debe configurar el plugin de Jenkins. Para ello se debe acceder a Jenkins > Home > Manage Jenkins > Global Tool Configuration, en el apartado SonarQube Scanner, para especificar la distribución del escáner.

  • La prueba de Análisis Estático la realizaremos con una tarea de tipo Pipeline en Jenkins. El proyecto a analizar debe contar con una configuración en base a propiedades, en un archivo sonar-project.properties o declarar en la línea del comando las propiedades requeridas. Para este ejemplo utilizaremos la opción de propiedades por línea de comando.
Tip:
El archivo sonar-project.properties puede generarse a tiempo de ejecución con Jenkins, mediante el paso para Pipelines writeFile.

# Archivo sonar-project.properties
# Clave única para una instancia de SonarQube
sonar.projectKey=mi:proyecto

# Nombre y versión (obligatorio para versiones previas a SonarQube 6.1)
sonar.projectName=Mi projecto
sonar.projectVersion=1.0
 
# Path para realizar el análisis relativo al archivo sonar-project.properties.
sonar.sources=.

# Codificación de código fuente
#sonar.sourceEncoding=UTF-8


sonar-scanner \
    -Dsonar.projectKey=mi:proyecto\
    -Dsonar.projectName="Mi projecto"\
    -Dsonar.projectVersion=1.0\
    -Dsonar.sources=.\
    -Dsonar.login=valor_de_token
Importante:
Es necesario haber inicializado el servidor de SonarQube.

node {
  stage('Código fuente') {
    git 'git@github.com:usuario/mi-proyecto.git'
  }
  
  stage('Análisis SonarQube') {
    // Nombre de la instalacion configurada en el Global Tool Configuration
    def scannerHome = tool 'Sonar-Scanner 2.8';
    
    // Se hace referencia por nombre al servidor configurado en Configure System
    withSonarQubeEnv('SonarQube 6.2') {
      // SONAR_AUTH_TOKEN = "Server authentication token" en la configuración de servidor
      sh "${scannerHome}/bin/sonar-scanner \
          -Dsonar.projectKey=mi:proyecto\
          -Dsonar.projectName=\"Mi projecto\"\
          -Dsonar.projectVersion=1.0\
          -Dsonar.sources=.\
          -Dsonar.login=${SONAR_AUTH_TOKEN}"
    }
  }
}


  • Si todo se encuentra correctamente configurado, en la consola de ejecución obtendrás una salida parecida a la siguiente, así como también podrás acceder a los resultados del análisis para su revisión.

 

Utilización de SonarQube como servicio

SonarQube ahora te ofrece una alternativa de análisis como servicio para proyectos Open Source. De esta manera solo necesitas tener configurado el ambiente de ejecución Sonar-Runner y hacer referencia a dicho servidor tanto en el Jenkins, como al momento de ejecutar el análisis dentro de las propiedades del proyecto.

  • Primero necesitas accesar al servicio y tienes dos opciones a través de tu cuenta GitHub o con un usuario en SonarQube. Para este caso utilizaremos la cuenta GitHub.

  • Debemos seguir los mismos pasos mencionados anteriormente para la generación de token de autenticación.

  • Necesitamos configurar este nuevo servidor en Jenkins (Configure System) con la URL https://sonarqube.com y el token generado.

  • Hacemos unas modificaciones al script que ya se tenia para la ejecución de la prueba, asignando la URL mediante variable de entorno, así como un identificador de tu organización.

Tip:
Si la autenticacion la realizaste por GitHub, por default se genera una organización usuario-github

// ...

withSonarQubeEnv('SonarQube (Online)') {
  sh "${scannerHome}/bin/sonar-scanner \
      -Dsonar.projectKey=mi:proyecto\
      -Dsonar.host.url=${SONAR_HOST_URL} \
      -Dsonar.sources=.\
      -Dsonar.organization=identificador_de_tu_organización \
      -Dsonar.login=${SONAR_AUTH_TOKEN}"
}

// ...


  • Ejecutamos el proyecto. Si la ejecución del análisis fue exitosa, podrémos ver los resultados en la URL https://sonarqube.com/dashboard/index/mi:proyecto.

 

Consecuencias

  • El Análisis Estático te permite detectar vulnerabilidades y en general te ayuda a conseguir una mejor calidad en tu código. Dicha calidad resulta importante desde la perspectiva de lineamientos de desarrollo e inclusive para lograr una evaluación más acertada durante pruebas de penetración.

  • Es necesario considerar que el alcance del análisis depende de la tecnología empleada en el código y que la detección para cada tecnología está directamente asociada con los plugins instalados en el servidor de SonarQube (algunos requieren comprar licencia). En este sentido SonarQube como servicio te ofrece una gama más amplia en cuestión a plugins instalados.
