Despliegue de Sitio Web en Jekyll con Jenkins

27 Oct 2016 . categoría: deployment . Comentarios
#ci #cd #jenkins #nginx #github #jekyll

Muy posiblemente alguien tendrá la misma inquietud por poner un Blog y quizá hasta coincida en la fórmula Jekyll-Nginx-GitHub. ¿En qué consiste? desarrollas un Sitio Web Estático con Jekyll y tienes tres posibles alternativas para publicarlo: levantas un servidor con Nginx y despliegas tu Sitio Web o generas un repositorio en GitHub y publicas tu Sitio en GitHub Pages o ambas (Nginx para el acceso principal y GitHub para el acceso con alta disponibilidad). Ahora que si crees que tener dos medios para publicar tu Sitio es mucho mantenimiento ¡para nada! un flujo de Jenkins puede ayudar a evitar ese esfuerzo.

 

  • Intención
  • Motivación
  • Aplicabilidad

Establecer un proceso de despliegue para tu Sitio Web (Jekyll) con Jenkins, sobre un servidor Nginx y GitHub Pages.

Cuentas con un servidor y has adquirido un dominio para exponer las particularidades de un proyecto o simplemente estás escribiendo un Blog. Adicional, consideras que es conveniente utilizar tu servidor para tareas más hayá de alojar tu Sitio. Entonces puedes tener tu servidor como punto de acceso principal de tu Sitio y en caso de alguna catástrofe con el servidor (o poco cuidado con los experimentos) tener expuesta tu página en un servicio dedicado como GitHub Pages.

Si quisieras que en cada cambio al repositorio principal de tu proyecto, los cambios se vieran reflejados en el Sitio, sin importar que el despliegue tuviera más de un destino, podrías pensar en el diseño de un proceso mediante Jenkins.

  • Automatización de despliegue de Sito Web en Nginx.
  • Automatización de despliegue de Sito Web en GitHub Pages.

 

Estructura

  1. Un desarrollador realiza commit de sus cambios y los centraliza mediante un push.
  2. GitHub detecta el push efectuado y genera un evento de push.
  3. Se ejecuta un proceso con el evento de push que realiza dos despliegues: uno sobre un servidor Nginx y otro sobre GitHub Pages.

 

Implementación

Prerrequisitos

  1. Tener configurada la ejecución de Jenkins mediante push.
  2. Tener configurado Ruby y Jekyll.
  3. Tener instalado Environment Injector Plugin.
Importante: Ambiente
Ubuntu 16.04.1 LTS alojado en DigitalOcean.
Jekyll 3.2.1
Nginx 1.10.0

 

Despliegue en GitHub Pages

Un despliegue en GitHub Pages se realiza a partir de un repositorio en GitHub. En este sentido, puedes tener, un Sitio general y una página Web por cada uno de tus proyectos en GitHub. El Sitio general lo obtienes generando un repositorio nuevo en GitHub con una característica, que el nombre se conforme de tu usuario seguido del dominio github.io, teniendo algo como: tu_usuario.github.io. Cualquier contenido agregado dentro del branch master será automáticamente reflejado como contenido Web. Al repositorio general lo accesas mediante https://tu_usuario.github.io/.

Por otra parte, cada repositorio que tengas en GitHub tiene la capacidad de exponer a Web. Para esto utiliza un branch con el nombre gh-pages. Todo el contenido dentro del branch gh-pages será publicado en Web. El acceso es mediante https://tu_usuario.github.io/tu_repositorio (independientemente de que tengas o no un repositorio general para tu usuario en GitHub).

Por lo tanto, el despliegue para GitHub Pages se realiza mediante una operación de push al branch master si se trata de una página personal (por usuario) o un push al branch gh-pages si se trata de una página para un repositorio del usuario (sub-página de usuario).

Tip:
Puedes ver la siguiente referencia para conocer más a detalle la generación y publicación de un Sitio personal con GitHub [1].

// Todos los cambios sobre archivos con seguimiento o sin seguimiento
git add -A
git commit -m "Actualización a contenido Web."


// Push a página general por usuario GitHub
git push origin master


// Push a página por repositorio GitHub
git push origin gh-pages

 

Despliegue en Nginx

Para realizar un despliegue en Nginx necesitas técnicamente de dos cosas, un archivo de configuración para las peticiones dirigidas al recurso que quieres hacer público y el contenido de tu recurso con el formato disponible para su publicación. De este modo, los pasos para el despliegue se resumen en lo siguiente:

  • Configurar el acceso a tu recurso. La configuración más elemental para un recurso en Nginx necesita definir cuál será la URL base (dominio, IP:puerto) de todas las peticiones y dónde se encuentra el contenido con el que se responderá la solicitud.
Nota:
Es posible exponer mediante Nginx: aplicaciones, contenido estático, imágenes, audio, video, archivos.
Nota:
Si tienes un dominio y necesitas direccionarlo a tu servidor, quizá pueda servirte esta configuración.

server {
  // Puerto para peticiones HTTP
  listen 80;
  listen [::]:80;

  // Configuración para escuchar peticiones tusitioweb.com
  server_name tusitioweb.com;

  // Ubicación física del contenido de tu Sitio Web
  root /var/www/tusitioweb_com;
  // Página principal
  index index.html;

  // Verificación y redireccionamiento interno hacia el
  // recurso solicitado
  location / {
    try_files $uri $uri/ =404;
  }
}


  • Si necesitas agregar SSL a tu configuración y forzar que tus peticiones sean HTTPS considera el siguiente ejemplo.

server {
  // Puerto para peticiones HTTP
  listen 80;

  // Configuración para escuchar peticiones tusitioweb.com
  server_name tusitioweb.com;

  // Redireccionamiento a petición HTTPS
  return 301 https://$host$request_uri;
}

server {
  // Configuración para escuchar peticiones tusitioweb.com
  server_name tusitioweb.com;

  // Puerto para peticiones SSL (HTTPS)
  listen 443 ssl default_server;
  listen [::]:443 ssl default_server;

  // Activación del SSL
  // Necesitas un certificado y su llave
  ssl on;
  ssl_certificate /etc/nginx/ssl/tusitioweb_com.crt;
  ssl_certificate_key /etc/nginx/ssl/tusitioweb.com.key;

  // Ubicación de Log de acceso y error
  access_log /var/log/nginx/tusitioweb_com.access.log;
  error_log /var/log/nginx/tusitioweb_com.error.log;

  // Ubicación física del contenido de tu Sitio Web
  root /var/www/tusitioweb_com;
  // Página principal
  index index.html;

  // Verificación y redireccionamiento interno hacia el
  // recurso solicitado
  location / {
    try_files $uri $uri/ =404;
  }
}


  • Aunque es posible tener un solo archivo de configuración, lo más adecuado sería tener un archivo de configuración por cada recurso que necesites exponer (ej. tu Sitio Web). El o los archivos de configuración los puedes trabajar en el directorio /etc/nginx/sites-available, una vez listos genera un enlace simbólico del archivo de configuración en el directorio /etc/nginx/sites-enabled/. Reinicia el servidor para los cambios se puedan reflejar.

sudo ln -s /etc/nginx/sites-available/tusitioweb /etc/nginx/sites-enabled/


sudo service nginx restart


  • Para el despliegue, debes copiar el contenido de tu Sitio Web en el directorio que especificaste como root en tu archivo de configuración. Si necesitas reemplazar el contenido, primero elimina los archivos anteriores y copia la última versión de tu contenido.
Nota:
El directorio genérico para despliegue Web en Ubuntu es /var/www.

rm -rf /var/www/tusitioweb/*


// Copia desde la localización altual en directorio
cp -a * /var/www/tusitioweb/

 

Asignación de permisos para usuario Jenkins

Muy posiblemente y debido a la configuración de seguridad, tu usuario jenkins no pueda líbremente manipular el contenido del directorio /var/www/ (si es que lo utilizaste como directorio raíz). Así que necesitamos asignar algunos permisos para que con el usuario jenkins se pueda trabajar apropiadamente.

La manera más fácil de administrar configuración de seguridad es mediante la gestión de grupos. Para este caso, el grupo por defecto que tiene privilegios sobre el contenido Web en Ubuntu/Debian es www-data. Por lo tanto necesitamos asignar los permisos necesarios para que el grupo pueda manipular el contenido del directorio /var/www/ [2].

  • Primero debemos agregar nuestro usuario jenkins al grupo www-data.

sudo usermod -a -G www-data jenkins


  • Agregamos los permisos de lectura y escritura para el grupo www-data sobre el directorio /var/www/.

sudo chgrp -R www-data /var/www
sudo chmod -R g+rw /var/www

 

Automatización de despliegue de Sitio Jekyll

Aún cuando la operación para despliegue tanto de Nginx como para GitHub Pages resulta poco laboriosa, el estar escribiendo los comandos y cambiando entre branches es una tarea repetitiva que debiera automatizarse. Nada mejor como delegar estas tareas a un experto en despliegue: Jenkins.

  • Primero necesitas tener tu entorno listo, esto significa tener un repositorio en GitHub (para este ejemplo utilizaremos un repositorio general), configurar la ejecución de Jenkins por push y tener instaladas las gemas necesarias para construir tu Sitio Web con Jekyll. Una versión nueva de tu Sitio se genera con los siguiente comandos:

// JEKYLL_ENV=production solo si tienes definido el contenido por ambiente
JEKYLL_ENV=production jekyll build


  • Posteriormente genera un Pipeline en Jenkins. El proceso definido con el archivo Jenkinsfile deberá contener el siguiente diseño:
    • Construir la nueva versión del Sito Web.
    • Almacenar temporalmente el contenido del Sitio Web.
    • Posicionarte en el branch de despliegue para GitHub (master o gh-pages).
    • Reemplazar el contenido del branch de despliegue con el contenido generado para la nueva versión del Sitio.
    • Reemplazar el contenido del Sitio Web para Nginx.
    • Generar el push hacia el branch de despliegue en GitHub.
  • Para la construcción necesitas hacer referencia a la distribución de Jekyll, asignando la variable de entorno para las gemas de Ruby.

node {
  // Branch con el fuente del Sitio Web
  git branch: 'personal', url: 'git@github.com:tu_usuario/tu_usuario.github.io.git'

  // Debes conocer la ubicación exacta de Ruby
  withEnv(["PATH+RB_SHIMS=/home/usuario/.rbenv/shims"]) {
    // En este bloque es posible ver todas las gemas de Ruby
    sh 'JEKYLL_ENV=production jekyll build'
  }
}


  • Dentro de un flujo de Jenkins es posible almacenar de manera temporal archivos, mismos que puedes utilizar a lo largo del proceso en cualquiera de los nodos sobre los que estés trabajando (¡esto es una maravilla!). Para ello utilizas un paso stash e identificas ese contenido con un nombre. Con dicho paso guardaremos temporalmente el contenido del Sitio construido con Jekyll.

node {
  // ...

  // Te ubicas en el directorio generado para el Sitio Web
  dir('_site') {
    // Guardas todo el contenido de manera temporal
    stash name: '_site', includes: '**'
  }
}


  • Como el despliegue en GitHub es mediante un branch, necesitamos colocarnos en el branch de donde se tomará el contenido a publicar. Debido a que todo el contenido del Sitio se genera en la construcción, la operación se traduce a borrar lo actualmente incluido en el branch de despliegue y colocar todo contenido guardado en temporal.

node {
  // ...

  // Creas otro workspace para no mezclar código fuente
  ws {
    // Branch para el deploy del Sitio Web
    git branch: 'master', url: 'git@github.com:tu_usuario/tu_usuario.github.io.git'

    // Se realiza la eliminación del contenido en el branch
    sh 'rm -rf *'

    // El contenido en temporal pasa se convierte en el nuevo contenido del branch
    unstash '_site'
  }
}


  • Una vez obtenido el contenido para la versión actualizada del Sitio, procedemos a realizar el despliegue en Nginx. Que como se mencionó anteriormente solo necesitamos reemplazar el contenido actual por el contenido construido.

node {
  // ...

  ws {    
    // ...

    // Se elimina el contenido actual del Sitio
    sh 'rm -rf /var/www/tusitioweb/*'
    // Se copia todo el contenido nuevo a la directorio del Sitio
    sh 'cp -a * /var/www/tusitioweb/'
  }
}


  • Por último, efectuamos el commit de los cambios referidos al contenido construido en el branch de despliegue. Este despliegue lo tomaremos como de “alta disponibilidad”, ya que ninguno de nuestros experimientos (excepto los push) pueden alterar o atentar la estabilidad del Sitio.
Peligro: Ejecuciones infinitas
Necesitas un mecanismo que evique que el push que realizarás en GitHub para el despliegue en GitHub Pages no genere un ciclio infinito de construcciones. Recuerda que la tarea de Jenkins se ejecuta a partir de un evento de push.

node {
  // ...

  ws {    
    // ...

    sh 'git add -A'
    sh "git commit -m \"Actualización a contenido Web.\""
    sh 'git push origin master'
  }
}


  • El Jenkinsfile completo tendría el siguiente aspecto (le puse unos adornos stage para identificar las actividades de ejecución):

node {
  stage('Construcción del Sitio') {
    git branch: 'personal', url: 'git@github.com:tu_usuario/tu_usuario.github.io.git'

    withEnv(["PATH+RB_SHIMS=/home/usuario/.rbenv/shims"]) {
      sh 'JEKYLL_ENV=production jekyll build'
    }
  }

  stage('Almacenamiento temporal') {
    dir('_site') {
      stash name: '_site', includes: '**'
    }
  }

  ws {
    stage('Preparación de despliegue') {
      git branch: 'master', url: 'git@github.com:tu_usuario/tu_usuario.github.io.git'

      sh 'rm -rf *'

      unstash '_site'
    }

    stage('Despliegue en Nginx') {
      sh 'rm -rf /var/www/tusitioweb/*'
      sh 'cp -a * /var/www/tusitioweb/'
    }

    stage('Despliegue en GitHub Pages') {
      sh 'git add -A'
      sh "git commit -m \"Actualización a contenido Web.\""
      sh 'git push origin master'
    }
  }
}

 

Consecuencias

  • Como se comentó anteriormente, considera que los push producen las ejecuciones de Jenkins. Si tu incluyes un despliegue a GitHub Pages desencadenarás un ciclo de ejecución-push-ejecución-push… infinito. Se puede pensar en un mecanismo que funcione a partir del reconocimiento de los commits que se realizan.

  • Otro paso adicional y muy importante puede ser la recuperación de errores o un plan de contingencia en el cual no se completarán los despliegues. Posiblemente el paso stash de Jenkins lo manipules de manera que en temporal almacenes el último contenido correcto y mediante manejo de errores puedas recuperes el despliegue anterior.

  • Podrías enriquecer el proceso para que puedas notificar sobre los eventos de tu importancia, respecto a los despliegues realizados. Piensa en que tienes medios como la integración de alguna herramienta de comunicación.
